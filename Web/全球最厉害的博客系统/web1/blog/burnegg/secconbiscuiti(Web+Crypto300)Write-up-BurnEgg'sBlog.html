seccon biscuiti (Web + Crypto 300) Write-up

发表于 2016-12-16   |   分类于 crypt , ctf , web   |   暂无评论

赛后去谷歌的时候找到了tinduong大牛 的writeup，很厉害。然后查了资料终于完全弄懂的cbc模式的两种攻击（见上一篇blog）。 今天才想起把前几天的分析记录下来。
首先，题目是提供源码的，主要index.php内容如下

function auth($enc_password, $input) {
    $enc_password = base64_decode($enc_password);
    $iv = substr($enc_password, 0, 16);
    $c = substr($enc_password, 16);
    $password = openssl_decrypt($c, ENC_METHOD, ENC_KEY, OPENSSL_RAW_DATA, $iv);
    echo $password;
    return $password == $input;
}

function mac($input) {
    $iv = str_repeat("\0", 16);
    $c = openssl_encrypt($input, ENC_METHOD, ENC_KEY, OPENSSL_RAW_DATA, $iv);
    return substr($c, -16);
}

function save_session() {
    global $SESSION;
    $j = serialize($SESSION);
    $u = $j . mac($j);
    setcookie("JSESSION", base64_encode($u));
}

function load_session() {
    global $SESSION;
    if (!isset($_COOKIE["JSESSION"]))
        return array();
    $u = base64_decode($_COOKIE["JSESSION"]);
    $j = substr($u, 0, -16);
    $t = substr($u, -16);
    if (mac($j) !== $t)
        return array(2);
    $SESSION = unserialize($j);
}

通读一遍，无非是个认证的流程。通过sqlite数据库搜索用户名得到用户hash解密与提交的密码对比，认证成功后cookie里保存了这段序列化数据和这段序列化数据加密后的最后32为作为mac校验防篡改。
然后很明显的，有个sqlite数据库注入，可以拿到数据库里的密文，然而这里的密文是在没有密钥的情况下解密不了的。所以想到通过sql union注入出自定义的一个用户。又发现代码在判断的时候是 $password == $input ，存在php的弱类型绕过。于是让post数据为username=adnx'union select '123','456'--&password;=0 注入出一个用户名为123 密码密文为456的用户，密文456在解密的时候失败，返回null。
而null==0 是为true的。成功登录。然而代码里还需要判断isadmin=1才会include flag文件
然后遇到坑点了，sql语句中就没有查询isadmin这个列，无论我们怎么构造sql，最后session数组里isadmin都是null。
先看一下现在的cookie里有什么

然后的目标很明显了，通过密码学攻击得到一个能被反序列化的字符串的密文，且反序列化过后s:7:"isadmin";N;变成s:7:"isadmin";b:1;
怎么去做呢？当时完全没搞懂cbc字符反转攻击的我以为是字符反转，比赛完了才发现，如果要反转最后一个block，就一定要毁坏倒数第二个block，所以应该尽量让最后一个block短，使得被毁坏的block放在数组的用户名里面才行。然而最短的时候也必须要";s:7:"isadmin";a:1;这20个字符才能成功反序列化。于是思路就断了。
直到赛后看到俄罗斯大牛tinduong的writeup加上完整分析了cbc加密算法。
发现代码里是有padding oracle attack的。通过union注入出不同密文，而代码调用解密的时候不成功返回null，同时使得password=0就能成功登录，反之则登录失败。
于是我们就能通过不停地尝试iv，来得到密文的中间值。得到中间值加上已知的明文，就能得到上一个block的密文。再对上一个block的密文进行padding oracle攻击，我们就能完整地还原出整段密文。（再次推荐上一篇blog）
那么回到这道题目。由于cbc加密的公式 Ciphertext-N= Encrypt(Plaintext XOR Ciphertext-N-1)
我们希望得到改变明文后新的密文。最好的办法就是让改变的东西放在最后一个block。这样在已知C4 (padding oracle attack)的情况下，可以更方便地构造出C5′(为了方便描述，借（chao）鉴（xi）一下大牛的表示方法)

P1 -> a:2:{s:4:"name"; -> aaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaa (C1)
P2 -> s:26:"xxxxxxxxxx -> bbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbb (C2)
P3 -> xxxxxxxxxxxxxxxx -> cccccccccccccccccccccccccccccccc (C3)
P4 -> ";s:7:"isadmin"; -> dddddddddddddddddddddddddddddddd (C4)
P5 -> N;}<PAD__PKCS#7> -> eeeeeeeeeeeeeeeeeeeeeeeeeeeeeeee (C5, MAC)

P1 -> a:2:{s:4:"name"; -> aaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaa (C1)
P2 -> s:26:"xxxxxxxxxx -> bbbbbbbbbbbbbbbbbbbbbbbbbbbbbbbb (C2)
P3 -> xxxxxxxxxxxxxxxx -> cccccccccccccccccccccccccccccccc (C3)
P4 -> ";s:7:"isadmin"; -> dddddddddddddddddddddddddddddddd (C4)
P5′-> b:1;}<NEW___PAD> -> <_dont_know_the_correct_cipher_> (C5′, NEW MAC)

为了得到新的密文，只能通过代码中唯一调用了加密函数的mac校验中来。
对mac校验的参数整理一下，发现我们现在可以通过union注入出来不同的name变量得到不同反序列化字符串的完整密文。
目标是得到C5′,而C5′ = Encrypt(P5′ ^ C4)
Encrypt的参数中，我们可控的只有部分明文，所以必须得延长这部分明文长到一个完整的block才行。（这就是为什么这位大牛构造了一个长达26的name变量）在这里P3就完全可控了。
由公式 C3=Encrypt(P3 ^ C2) 我们控制新的P3′=C4 ^ C2 ^ P5′
那么新的C3′=Encrypt(P3′ ^ C2)=Encrypt(C4 ^ C2 ^ P5′ ^ C2)=Encrypt(C4 ^ P5′)=C5′ 这就是我们一直想要的新明文的mac了，仍然通过padding oracle 还原出来即可。
