Base64编码

火日攻天

    Base64可以说是最常见的编码之一，需要重点的了解。最权威的说明详见
    rfc4648（Standard 'base64' encoding），以及Base64的起源 rfc2045（Base64 transfer encoding for MIME）

    定义

    Base64是一种基于64个可打印字符来表示二进制数据的表示方法（binary-to-text encoding）

    两个重点：
    • 编码的对象：二进制（任何输入，要先转成二进制）
    • 用于编码的字符：64个可打印字符
    why 64个？

    其实理论上来说，2个足矣（01二进制流），不过编码追求的是简单高效，所以最好的做法是直接将二进制分组，按组编码，而分组编码时少用1bit就是32个可打印字符，报文膨胀的太厉害；多1bit就需要128个，可打印字符不够用啊！所以用6bit为一个单位编码，2^6=64个

    起源

    Base64编码源自MIME content transfer encoding，而MIME就是多用途互联网邮件扩展（Multipurpose Internet Mail Extensions）。

    而诞生MIME的原因就是早期电子邮件标准并不允许在邮件消息中使用7位ASCII字符集以外的字符，直到现在电子邮件依然是基于字符的而不是二进制的，这就导致了一些非英语字符消息和二进制文件，图像，声音等非文字消息都不能在电子邮件中传输。

    所以怎么解决？编码！把二进制编码成可见字符，就可以在电子邮件中传输了，接收端在解码为二进制即可。

    当然，MIME规定的编码方式不止Base64，还有7bit（UTF-7），quoted-printable等，只不过大部分人只了解Base64而已。

    编码方式

    由于Base64使用小字符集编码大字符集，必然会带来报文长度的增加，而Base64一般被认为会平均增加33%的报文长度。

    由于Base64使用64个可打印字符编码，而2^6=64，即对二进制流而言，每6bit可以编码为64个可打印字符中的一个。在计算机系统中信息计量单位是字节（a unit of digital information ），即8bit，取个公倍数很容易得出，我们可以把3字节（24bit），拆成4个6bit，每一个6bit代表一个可打印字符，而每一个可打印字符其实占1字节（8bit），故Base64是用4个字符（字节）编码3个字符（字节）来实现编码后不出现不可打印字符，代价是报文长度增加(4-3)/3=33%

    编码步骤

    • 首先在二进制数据后填充若干个0，按照6bit对齐
    • 将二进制数据按照6bit切分为若干段，计算每一段二进制代表的数值。如下图切分后的第一段是010011，即十进制的19
    • 查表，根据数值查找其代表的可见字符，如上图第一段的19，查表可得T，所以第一段被编码为了T，以此类推文本MAN的Base64编码为TWFu
    • 用等号（=）填充编码后的字符串，使得字符串按照4个字符对齐
    注意
    • 虽然说包括了等号共有65个字符，但是等号的个数是可以推测出来的（4字符对齐），所以其实可以不传输等号。（不过有些base64的库可能不允许这种没有等号填充的输入）
    • Base64编码后都是可见字符，但是不代表我们可以在任何地方直接传输编码后数据，比如HTTP的GET，POST，COOKIE，都要将加号、等号进行url编码后才能传输
    • Base64编码还是很容易判断的，一长串字符串只有[A-Za-z0-9+-]，并且常常以=结尾，都是Base64标志性特点。有时候看到一长串AAAAAA也可以作为判断的依据，因为A对应的是二进制000000，而在文件中，是经常存在大段空字节的。

    换行问题

    实际使用中会发现有些库在输出base64编码后的字符串时，会每隔若干个字符(76个字符)后添加一个换行符（比如Linux Shell）。这其实是rfc2045中规定的（The encoded output stream must be represented in lines of no more than 76 characters each.）。这是为了照顾古老的语言和古老的程序，比如MIME面对的基本都是C编写的SMTP服务器，而他们可能一行的缓冲有限，无法在一行接收过多的数据，在解码时，根据rfc2045中的规定，解码器会自动忽略其他字符，包括换行符，所以输入是否换行并不影响解码。

    所以其实换行的原因与Base64没有关系，而是与MIME使用场景有关，所以在后来Base64标准（rfc4648）中并未提到需要换行，而是指出依场景而定，所以在越高级的语言，比如Python，支持无限长的输入，其编码后的输出的就是无换行的字符串。

    由于换行的影响，Base64实际平均增加37%的报文长度。

    “字符串编码”问题

    在使用Base64库时，有时能看到输入参数中有一个编码的选项，其实这个和Base64并没有关系，原因是Base64编码的是二进制，如果输入是一个字符串，就必须先转换为二进制数据，而字符串转为二进制或者二进制转为字符串就涉及到字符串编码问题了，而这些输入参数指的就是字符串编码。

    比如同样编码字符串“火日攻天”

    • GB2312：u/DI1bmlzOw=
    • UTF-8: 54Gr5pel5pS75aSp

    解码

    解码就是编码的逆过程，一般会根据输入是否为4字符倍数判断是不是正常的输入，然后去掉等号填充，查表得出对应得二进制流，最后按照8bit对齐去除多余填充的0即可。唯一需要注意的是，根据rfc2045，输入中所有非编码字符均会被忽略，不影响解码。

    实现

    几乎所有语言都有Base64编码的库或者函数，这里以编解码firesun字符串为例子

    Linux Shell

    • 编码
    echo -n "firesun"|base64
    • 解码
    echo "ZmlyZXN1bg=="|base64 -d

    注：参数-n表示在最后不输出换行符

    JavaScript

    • 编码
    btoa("firesun")
    • 解码
    atob("ZmlyZXN1bg==")

    注：仅支持ASCII码

    Python

    • 编码
    import base64  
base64.b64encode("firesun")
    • 解码
    import base64  
base64.b64decode("ZmlyZXN1bg==")

    注：

    • base64.encode()可以用来编码文件
    • base64.encodestring()与base64.b64encode()都可以用来编码字符串，区别是b64encode还可以指定将+/替换成别的字符串

    Ruby

    • 编码
    require "base64"  
encoded = Base64.encode64('firesun')
    • 解码
    require "base64"  
decoded = Base64.decode64('ZmlyZXN1bg==')

    PHP

    • 编码
    echo base64_encode("firesun");
    • 解码
    echo base64decode("ZmlyZXN1bg==");

    Android

    • 编码
    import android.util.Base64;  
String encoded = Base64.encodeToString("firesun".getBytes(), Base64.DEFAULT));
    • 解码
    import android.util.Base64;  
String decoded = new String(Base64.decode("ZmlyZXN1bg==", Base64.DEFAULT));

    注：

    • getBytes()和new String()时都可以显示指定字符串编码的（详见Java8实现），否者采用默认字符串编码。
    • Base64.DEFAULT代表的输出会换行，需要输出不换行的需要使用Base64.NO_WRAP

    Java8

    • 编码
    import java.util.Base64;  
String encoded = Base64.getEncoder().encodeToString("firesun".getBytes("utf-8"));
    • 解码
    import android.util.Base64;  
String decoded = new String(Base64.getDecoder().decode("ZmlyZXN1bg=="), "utf-8");

    注：

    • 可以用Base64.getMimeEncoder().encodeToString("firesun".getBytes("utf-8"))来输出带换行的编码结果

    类似的编码方案

    Base64url

    全称Base 64 Encoding with URL and Filename Safe Alphabet，详见rfc4648，适用于URL和文件名等场景，使用减号与下划线代替了+/，防止编码后出现有歧义的符号(用于padding的等号并不是必须的，所以也不会造成影响)，也就不需要二次编码即可直接使用，在一些网页中有时能够见到形如“amF2YTg_amF2YTg=”的字符串，就是很明显的Base64url，在Java8中可以使用 Base64.getUrlEncoder().encodeToString("firesun".getBytes("utf-8"))来调用Base64url编码

    Base32

    32个可见字符编码，原理大同小异，编码字符[A-Z2-7]，=用于填充，最大的特点是只有大写没有小写，并且有可能出现2个以上的等号填充。以Python为例： 

    import base64  
base64.b32encode("firesun")  
base64.b32decode("MZUXEZLTOVXA====")

    Base16

    别被名字骗了，这不就是数据16进制表示么（Hexadecimal），同样以Python为例（多种写法）：

    import binascii  
binascii.b2a_hex("firesun")  
binascii.b2a_hex(u"火日攻天".encode("utf-8"))  
u"火日攻天".encode("utf-8").encode('hex')  
binascii.a2b_hex("6669726573756e")  
print 'e781abe697a5e694bbe5a4a9'.decode("hex").decode("utf-8")

    UTF-7

    同样是MIME的编码中的一种，UTF-7是一个修改的Base64（Modified Base64）。主要是将UTF-16的数据，用Base64的方法编码为可打印的ASCII字符序列。目的是传输Unicode数据。主要的区别在于不用等号"="补余，因为该字符通常需要大量的转义。
